Codeplay is known for implementing OpenCL™ and SYCL™ runtimes and compilers for their customers' acceleration platforms. Now we aim to target customers in the automotive sector, who want to run highly complex AI algorithms on their accelerators, which will provide advanced driving assistance (ADAS) and autonomous driving (AD) features to make our roads safer. To support these customers Codeplay needs to ensure that the code is sufficiently safe. The safety is ensured primarily by static analysis, manual review and testing. However, in anything but a trivial code base it is not possible to undertake these to 100% coverage, so a representative sample has to be chosen. Being able to predict which modules are likely to contain bugs helps direct these efforts to the places where they are most valuable.

Codeplay's software has been in development for many years and we have therefore accrued a lot of data in the form of bug reports, merge requests and git revisions. This data encodes historical information such as the prevalence of defects per module. By extrapolating this data we can estimate the number of bugs remaining in the software as well as predict which parts of the code are likely to contain more bugs.

Responsibilities

  • Performing data mining of our software repositories through the Jira issue tracker.
  • Creating scripts that call into Jira's API, to extract data such as the defect rate of different areas of code, the defect rate of code produced by different developers, etc.
  • Optionally, extracting additional metrics from the code, such as complexity or comment density, and try to establish a correlation between them and the defect rate.
  • Writing a blog post describing the techniques and some high-level results.
